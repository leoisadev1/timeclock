Design read:
- Reading this as an operational scheduling/time-attendance app for managers and hourly employees, with a calm command-center language, leaning toward shadcn/Tailwind, dense useful surfaces, restrained orange/ink accenting, and Linear/Vercel-grade product motion.

Compact UI brief:
- Surface: one responsive web app with manager workspace, employee PIN portal, and station/kiosk mode.
- User job: create/publish schedules, clock employees in/out, monitor today's attendance, and inspect scheduled-vs-actual time.
- Visual posture: calm restaurant operations command.
- Information architecture: app shell/location selector, Today dashboard, Schedule builder, Employees, Reports, Employee portal, Station route.
- Composition: split workspace, table/grid schedule, status rails, inspector/modals, kiosk PIN pad. Use cards only for repeated records or contained modals.
- Interaction: fast tab/route transitions, selected-location continuity, modal enter/exit, schedule cell hover/focus, punch action feedback, PIN keypad states, polling/update pulse.
- Forbidden patterns: landing-page hero, decorative bento grid, fake SaaS cards everywhere, purple/blue gradient blobs, giant marketing copy, one-note dark slate dashboard, hidden primary workflow below intro content.

Core screens/routes to build:
- /: useful demo entry with manager login shortcut, employee PIN entry, station entry, and current demo status. Keep it product/workflow-first, not a marketing landing page.
- /dashboard or /_auth/dashboard: manager shell with location selector and tabs/nav for Today, Schedule, Employees, Reports, Settings.
- /employee: PIN-based employee web view with schedule, current status, punch actions, today's events, recent history.
- /station: kiosk mode with manager unlock placeholder, location select, large PIN pad, employee confirmation/status, punch actions, exit confirmation.

Implementation guidance:
- Use shadcn-style components from packages/ui first: Button, Card only when appropriate, Checkbox, DropdownMenu, Input, Label, Skeleton, Sonner.
- Add missing shared components only when needed, for example badge, tabs, dialog, table, select, separator, textarea, avatar, tooltip, segmented control. Keep them in packages/ui/src/components.
- Use lucide-react icons because the repo already depends on it; keep stroke width consistent.
- Build a real app shell, not a landing page.
- Keep tables/grids readable on desktop and practical on mobile. On mobile, switch schedule grid to day tabs or a vertical agenda instead of horizontal squeeze.
- Use semantic status colors sparingly: green working, amber warning, red late/no-show, blue/neutral info, primary orange action.
- Keep type compact. No viewport-scaled fonts. No giant dashboard headings.
- Avoid React useEffect for derived state. Use render-time derivation, event handlers, TanStack/Convex hooks, or adapter functions.
- Do not add a new animation library unless already present or clearly justified. Prefer CSS transitions and tw-animate-css/shadcn motion primitives.

Motion system:
- Motion intensity: 3-4 for manager workflow, 4-5 for station/employee feedback.
- Use explicit transition properties, not transition-all.
- Button press: 70-120ms transform/opacity.
- Hover/focus/selected states: 120-180ms.
- Modals/drawers/popovers: 180-260ms, opacity + 4-12px translate from the trigger/edge.
- Schedule/status updates: subtle pulse or highlight once, then settle.
- Respect prefers-reduced-motion.
- No looping decorative motion near readable operational content.
